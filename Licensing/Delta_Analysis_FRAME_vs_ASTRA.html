1 · Structural Difference: How Products Are Organised

🟢 ASTRA – Two-Sheet Model

  • Sheet 1: "Shopping List" → Paid licenses only (148)
  • Sheet 2: "License inventory for FOSS" → Freeware & OSS (123)
  • Clear structural separation: commercial vs free
  • Aggregation Pivots provided (Vendors vs Products, Free or OSS)

🟠 FRAME – Single-Sheet Model

  • One sheet "Shopping_List_Vorbereitung" contains ALL 590 products
  • No structural separation of paid vs free/OSS
  • Free/OSS not found as a distinct Type value (0 rows explicitly labelled)
  • Relies on Type column — but 41% are "TBD" or blank
⚠ Risk for FRAME: Without explicit free/OSS categorisation, open-source products may be scheduled for procurement unnecessarily, or freeware compliance checks may be missed.

6 · Procurement Process Maturity

🟢 ASTRA – Inventory Model

  • Acts as a static inventory / BOM of software needed
  • No procurement workflow fields (no status, owner, deadline)
  • Notes field (22% fill) used for ad-hoc clarifications
  • Relies on external process for actual procurement scheduling
  • Two footnotes in the file define TBD handling rules (metric + type)

🟠 FRAME – Procurement Workflow Model

  • Acts as a living procurement tracker
  • Ready_for_Procurement: 100% filled — tracks stage per item
  • Priority_Date + Reason: 51% filled — urgency-driven scheduling
  • Changes since Feb 2025: 41% — delta tracking vs baseline
  • Action + Responsible: exists but underused (<5%)
  • Due_Date: column exists but 0% filled — process gap

8 · Summary: Key Behavioural Gaps & Recommendations

🔴
Gap 1 – FRAME lacks Free/OSS categorisation
ASTRA explicitly separates 123 freeware/OSS products. FRAME has zero. This creates compliance risk — freeware usage may not be tracked, and OSS licence obligations (GPL, MIT, etc.) may go unmanaged.
→ Recommendation: Add a "License_Category" column to FRAME with: Paid / Freeware / OSS
🔴
Gap 2 – FRAME has 60% unresolved License Type (TBD / blank)
241 FRAME products have no defined payment model. ASTRA classifies 100%.
→ Recommendation: Set a deadline to resolve all TBD entries before procurement cutover.
⚠
Gap 3 – ASTRA has no Procurement Workflow tracking
ASTRA has no procurement status, owner, or deadline fields. Procurement must be managed entirely outside the file.
→ Recommendation: Add Ready_for_Procurement, Responsible, and Due_Date columns to ASTRA — or use the merged Microsoft List as the joint tracker.
⚠
Gap 4 – FRAME Due_Date column is 0% populated
Despite having the column, no due dates are ever set — the priority timeline is captured only in the "Priority Date" field which is informal text.
→ Recommendation: Make Due_Date a mandatory field tied to Priority Date.
ℹ
Observation – 12 vendor type mismatches signal scope difference, not error
Most mismatches occur because ASTRA inventories ALL installed software (including free tools), while FRAME only tracks what needs to be procured. This is an intentional scope difference, not a data quality issue — but should be aligned in the shared Microsoft List.
✅
Strength – ASTRA's pivot analytics sheets
ASTRA provides 2 built-in pivot tables (Vendors vs Products, Free or OSS summary). FRAME has none.
→ The merged Microsoft List can replicate this with views and filters.
✅
Strength – FRAME's multi-region metric granularity
FRAME's metrics like "NNU Taiwan", "CN EU", "User Portugal" reflect a truly global deployment scope. ASTRA's simpler metric model ("user", "floating") doesn't support multi-region allocation.
→ In the merged Microsoft List, both models coexist — the Region dimension from FRAME metrics can be extracted into a dedicated column.
